# Кастомний Bot API сервер замість in-process TDLib

**Дата:** 2026-06-05
**Статус:** дизайн затверджено, очікує плану імплементації

## Мета

Прибрати з бота in-process TDLib (`tdl` + `prebuilt-tdlib`) і другу MTProto-сесію.
Замість цього — публічний форк `tdlib/telegram-bot-api` з невеликим патчем, який
додає методи, що їх бракує Bot API. Бот ходить на цей сервер за **все**: і
звичайні виклики, і колишні TDLib-довідки.

## Контекст

Зараз TDLib у боті використовується для:

| Виклик | Навіщо | Де |
| --- | --- | --- |
| `getMessages` | мультиквоти `/q <n>` + reply-зв'язка для флага `r` | `src/features/quote/select.ts` |
| `getUser` | преміум emoji-статус біля імені | `assemble.ts` → `getUserEmojiStatus` |
| `getChat` | довідка про чат/юзера (enrichHidden тощо) | `src/services/tdlib/index.ts` |

Bot API цих даних не віддає (`getChat` хмарного API працює лише для юзерів, що
стартували бота). Форк сервера має всередині той самий TDLib з тією ж
бот-сесією — отже може віддати все це нативно, вже в Bot API shape.

Розглянутий і відхилений варіант: **tdlight-telegram-bot-api** (готовий форк з
`getMessageInfo`). Причина відмови: відставання від upstream (Bot API 9.5 проти
10.0, docker-образи ~3 міс.), синк «хвилями» — не проходить вимогу
«надійно і актуально». Його корисні безпекові прапорці переносимо в наш патч.

## Рішення (огляд)

```
┌─────────────┐  HTTPS (Bot API + кастомні методи)   ┌──────────────────────────┐

┌─────────────┐  HTTPS (файли через сайдкар)          │  │ (наш форк, --local) │  │
```

## 1. Форк і патч

- Публічний форк: **`LyoSU/telegram-bot-api`** (уже створений, `master`
  синхронізований з upstream станом на 2026-06-05). Гілка **`custom`** =
  upstream `master` + один логічний блок комітів (Δ). Історія завжди `upstream + Δ`
  (rebase, не merge) — дельту видно цілком через
  `git diff upstream/master..custom`.
- Код патчу — максимально в окремих файлах; у `Client.cpp` лише гачки
  реєстрації. Мінімізує rebase-конфлікти.
- Усі додані фічі — **opt-in через прапорці**, у дефолті форк поводиться як
  upstream (форк публічний — має бути корисний і безпечний для будь-кого).

### Склад патчу

| Фіча | Вмикання | Походження |
| --- | --- | --- |
| `getMessages(chat_id, message_ids[])` — батч до 50, відповідь: масив `Message` (стандартна серіалізація сервера, включно з `reply_to_message`) | метод доступний завжди | наша |
| `getUserInfo(user_id)` — `User` + `emoji_status_custom_emoji_id` | метод доступний завжди | наша |
| Allowlist бот-токенів: запити з невідомим токеном відкидаються (включно з file-шляхами) | `--allowed-tokens=...` | наша; закриває «відкритий проксі» публічного сервера |
| Лише відносні шляхи файлів у local mode (захист від path traversal) | `--relative` | tdlight |
| Приховування токена/webhook URL на stats-сторінці | `--stats-hide-sensible-data` | tdlight |

## 2. Автоапдейт (GitHub Actions у форку)

Лінійний пайплайн, без PR і без тестів — запобіжник на боці деплою
(healthcheck у Coolify: контейнер не піднявся → лишається старий).

1. **Cron щодня**: `git fetch upstream`; нових комітів немає → exit 0.
2. Rebase `custom` на `upstream/master`. Конфлікт → workflow червоний,
   GitHub-нотифікація, нічого не пушиться. Єдиний ручний сценарій.
3. Збірка Docker-образу: clang + ninja + ccache (кеш у GH Actions cache +
   `cache-from/to: type=registry`). TDLib — git submodule, збирається в
   цьому ж білді. Перший білд ~60–90 хв, інкрементальні ~10–20 хв.
   Тільки `linux/amd64`.
4. Push у GHCR: `:latest` + `:<bot-api-версія>`. Multi-stage → фінальний
   образ ~15 МБ (alpine + статичний бінарник), non-root.
5. `curl` на Coolify deploy webhook (URL — тільки в Actions secrets).

## 3. Деплой (Coolify, окремий сервер)

Одна compose-аплікація, два сервіси + спільний volume:

- **bot-api** (наш образ): `--local --relative --allowed-tokens=<токен>
  --stats-hide-sensible-data`, env `TELEGRAM_API_ID`/`TELEGRAM_API_HASH`,
  порт 8081 → публічний HTTPS-домен через Coolify proxy (Let's Encrypt).
  Stats-порт не публікується (доступний лише в внутрішній мережі).
  Persistent volume на робочу директорію (сесія TDLib переживає редеплої).
  Healthcheck + авторестарт.
- **files** (caddy або nginx): той самий volume read-only, віддає файли по
  HTTPS на окремому домені/шляху. Авторизація — токен як сегмент шляху (та
  сама модель, що в хмарного API), без directory listing. Range-запити та
  sendfile — задарма від caddy/nginx.

Local mode → жодних лімітів на розмір файлів; bot-api процес не витрачає
себе на роздачу статики.

## 4. Зміни в боті (цей репозиторій)

- grammY: `apiRoot` з нового env `BOT_API_ROOT` (`src/config/env.ts`,
  `src/core/bot.ts`).
- **Видаляється** `src/services/tdlib/` цілком: клієнт, circuit breaker,
  `normalize.ts` (сервер віддає вже Bot API shape), типи `TdMessage` →
  залишковий структурний тип біля `select.ts`. З `package.json` ідуть
  `tdl`, `prebuilt-tdlib`; з env — `TELEGRAM_API_ID`, `TELEGRAM_API_HASH`,
  `DISABLE_TDLIB`; зникає `npm run tdlib:types`.
- Новий тонкий сервіс (HTTP-клієнт до нашого сервера) реалізує наявний
  інтерфейс `MessageFetcher` (`getMessages`, `isHealthy`) + заміну
  `getUserEmojiStatus` через `getUserInfo`. `select.ts` і graceful
  degradation (упав фетч → квота з одного повідомлення) не змінюються.
- LRU-кеші users/chats переїжджають у новий сервіс як були.

## 5. Інтеграція quote-api (окремий репозиторій, поза цим скоупом)

- Базовий URL завантаження файлів → домен сайдкара.
- Те саме для стрімінгу медіа в mini-app, якщо він качає по file_id.
- Виконати **до** перемикання токена: після logOut з хмари старі
  `api.telegram.org/file/...` URL перестануть працювати.

## 6. Безпека

- Allowlist токенів — обов'язковий у нашому деплої (сервер публічний).
- `--relative` — обов'язковий разом із `--local` (шляхи перетинають межі
  сервісів через volume).
- Контейнер: non-root, read-only rootfs крім робочого volume.
- Секрети: `api_id`/`api_hash`/токен — тільки в Coolify env; webhook URL —
  тільки в GitHub Actions secrets. У публічному репо — нічого.
- Опційне посилення на потім: IP-allowlist (Traefik-правило в Coolify) на
  сервери бота і quote-api.

## 7. Міграція

1. Підняти сервер + сайдкар на Coolify, перевірити `getMe` і кастомні
   методи вручну (своїм токеном).
2. Оновити quote-api на новий файловий URL (підготувати, не вмикати).
3. `logOut` у хмарного Bot API (інакше ~10 хв очікування при перемиканні).
4. Перемкнути бота: `BOT_API_ROOT` + видалений TDLib-код, деплой.
5. Моніторити: RAM сервера, латентність, метрики бота.
6. Відкат: прибрати `BOT_API_ROOT` (бот повертається в хмару після
   повторного логіну), quote-api — на старий URL.

## 8. Ризики

| Ризик | Мітигація |
| --- | --- |
| Сервер — критичний шлях усього бота (зараз TDLib опціональний) | healthcheck + авторестарт; ретраї grammY; graceful degradation мультиквот лишається |
| RAM: сервер тримає всі чати бота в пам'яті | моніторинг після міграції; запасні ходи — періодичний рестарт, черрі-пік memory-оптимізацій tdlight |
| Rebase-конфлікт патчу з upstream | патч ізольований в окремих файлах; червоний workflow → ручний фікс (рідко) |
| Мережа бот ↔ сервер (різні машини) | таймаути/ретраї grammY; обидва сервери — у одного провайдера за можливості |
